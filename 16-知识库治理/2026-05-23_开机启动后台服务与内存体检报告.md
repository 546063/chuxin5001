# 开机启动后台服务与内存体检报告

## 一句话结论

当前后台常驻主要由 AI 工具链、远程/NAS 隧道、代理网络、同步备份、微信和 Codex/Claude 会话构成；本轮只分析，不删除、不停用。

## 明确开机/定时启动项

| 启动项 | 类型 | 行为 | 判断 |
|---|---|---|---|
| `ai.openclaw.gateway` | LaunchAgent | `RunAtLoad` + `KeepAlive`，监听 `127.0.0.1:18789` | OpenClaw 网关常驻，AI 工具链核心 |
| `ai.openclaw.node` | LaunchAgent | `RunAtLoad` + `KeepAlive`，连接 OpenClaw 网关 | OpenClaw 节点常驻 |

## 安全备注

- `com.cc-connect.service` 的 LaunchAgent 环境变量中存在 API token 类敏感配置。本报告不记录具体值。
- 建议后续把敏感 token 从 plist 明文环境变量迁移到更安全的配置文件、Keychain 或权限更严格的本地 secret 文件。

## 不建议立即删除的项

- OpenClaw gateway/node：AI 自动化依赖。
- cc-connect：远程 IM 桥接依赖。
- NAS tunnel / mount：当前可能承担备份和远程访问。
- Obsidian backup / weekly supplement：知识库安全和增量治理依赖。

## 可进一步确认的项

| 项 | 原因 | 下一步 |
|---|---|---|
| `com.taozhichu.unified-nas-tunnel` vs `com.zerotier.macmini-ssh` | 都在做 SSH 转发，存在功能重叠可能 | 比对端口用途后决定是否合并 |
| `com.ai.douyin.juliang.fullupdate` | 每天运行，可能产生资源/网络开销 | 查最近日志，确认是否仍需要每日更新 |
| `com.google.GoogleUpdater.wake` | 每小时唤醒 | 如不影响可保留；若追求极简后台再评估 |
| `next dev` | 已运行 2 天以上 | 开发完成后可手动停止，但本轮不停止 |

## 下一步建议

1. 先查 OpenClaw、cc-connect、巨量更新、NAS 备份日志，确认是否有报错循环。
2. 对两个 SSH 隧道做端口用途表，判断是否可以合并。
3. 给 `food-machinery-hero` 的 dev server 建立“需要时启动”的规则，避免长期占用。
4. 迁移 LaunchAgent 中的敏感 token，不要继续明文放在 plist 环境变量里。
