# 投流知识库增量更新机制

## 一、新增知识录入流程

### 1.1 触发场景

| 触发源 | 处理方式 |
|--------|---------|
| 实际投放新数据 | 用 `[[20-Facebook投流实战数据记录模板]]` 回补 |
| 平台政策变更 | 更新合规清单 + 标记 `updated` 日期 |
| 新市场/新品类 | 在案例库/素材库追加条目 |
| 新获客链路 | 新增 SOP 文档，编号递增 |
| 客户反馈/复盘结论 | 更新对应 SOP 的"常见坑"章节 |

### 1.2 命名规范

- **SOP 文档**：`XX-平台+功能+SOP.md`（如 `25-Facebook落地页优化SOP.md`）
- **案例/素材库**：`XX-行业+平台+类型库.md`
- **工具/模板**：`XX-平台+功能+模板.md`
- 编号从 25 开始递增（24 是最后一个已有文档）

### 1.3 新文档 Frontmatter 必填项

```yaml
title: "文档标题"
aliases: ["别名1", "别名2"]
tags: [tag1, tag2]
type: "文档类型（SOP/模板/案例库/治理SOP）"
module: "所属模块"
platform: "平台名称"
routing_target: [target1, target2]
quality_level: "A"
summary_cn: "中文摘要"
summary_en: "英文摘要"
related: ["[[相关文档1]]", "[[相关文档2]]"]
retrieval_keywords: ["搜索关键词1", "搜索关键词2"]
decision_value: "决策价值——为什么需要这个文档"
created: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
```

## 二、路由表同步更新

### 2.1 新增文档后必做

1. **更新总索引**：在 `00-Facebook投流知识库总索引.md` 导航表中新增行
2. **更新路由入口**：在 `10-Facebook投流路由入口.md` 检查是否需要新路由规则
3. **更新路由表**：在 `16-Facebook投流常见问题路由表.md` 追加问题映射
4. **更新 JSON 路由**：在 `routing_rules_facebook_ads.json` 中：
   - 如需新 routing_target，在 `routes` 对象中新增条目
   - 如复用已有 routing_target，在 `preferred_notes` 中追加路径
   - 在 `trigger_keywords` 中补充关键词

### 2.2 JSON 路由文件维护 Checklist

- [ ] 运行 `python3 -m json.tool routing_rules_facebook_ads.json > /dev/null` 校验
- [ ] 检查所有 `preferred_notes` 指向的文件是否存在
- [ ] 确认 `trigger_keywords` 覆盖新文档的核心关键词
- [ ] 更新 `version` 字段为当前日期

### 2.3 全局路由主索引维护

- 如果新增整个平台的知识库，在 `routing_rules_unified_ads_master.json` 的 `platforms` 对象中新增条目
- 更新 `total_routes` 数字

## 三、定期自测流程

### 3.1 测试频率

| 测试类型 | 频率 | 负责人 |
|---------|------|--------|
| 路由准确率 | 每次新增/修改文档后 | AI Agent |
| 断裂链接检查 | 每周 | AI Agent |
| Frontmatter 完整性 | 每月 | AI Agent |
| 内容时效性 | 每月 | 人工 + AI |

### 3.2 路由准确率测试

使用 `[[Facebook投流路由测试问题集]]` 中的 100 条测试问题：

1. 随机抽 20 题
2. 检查每题是否命中正确的 `routing_target`
3. 检查是否能读取到对应文档
4. 检查回答是否包含核心信息
5. 目标通过率 ≥ 90%
6. 未通过的题：记录 → 修正路由 → 复测

### 3.3 断裂链接检查

```bash
# 检查所有 wikilink 是否指向真实文件
cd /Users/mac/ai-workspaces/AI-Workspace-Obsidian
# 提取所有 wikilink，检查目标文件是否存在
grep -roh '\[\[.*?\]\]' 03-海外投流/Facebook投流知识库/ | \
  sed 's/\[\[\(.*\)\]\]/\1/' | sort -u | \
  while read link; do
    found=$(find . -name "*${link}*" 2>/dev/null | head -1)
    if [ -z "$found" ]; then
      echo "BROKEN: $link"
    fi
  done
```

### 3.4 Frontmatter 检查清单

- [ ] `title` 非空
- [ ] `routing_target` 与 `routing_rules_*.json` 中的 key 一致
- [ ] `decision_value` 是字符串（不是数字或嵌套对象）
- [ ] `created` / `updated` 日期格式为 `YYYY-MM-DD`，年份正确
- [ ] `related` 中所有 wikilink 指向真实文件
- [ ] `quality_level` 为 `A` / `B` / `S` 之一
- [ ] `type` 与文档实际类型一致

## 四、内容时效性管理

### 4.1 需要定期更新的内容

| 内容类型 | 更新周期 | 说明 |
|---------|---------|------|
| 广告政策/合规清单 | 平台更新时 | Meta 政策变化不定，以官方为准 |
| 投放数据（CPM/CTR/CPL） | 投放后 | 从实战数据模板回补 |
| 平台功能/界面 | 季度 | Meta Ads Manager 界面常变 |
| 受众定向选项 | 季度 | 新增/废弃定向选项 |
| 素材规格 | 平台更新时 | Reels/Stories 尺寸可能变化 |
| 竞品分析 | 月度 | 用 Meta Ad Library 定期更新 |

### 4.2 过期标记

对于不确定时效性的内容，在文档顶部添加：

```markdown
> ⚠️ 本文档部分数据基于 XXXX 年 XX 月的平台状态，投放前请以 Meta 官方后台为准。
> 最后验证日期：YYYY-MM-DD
```

## 五、质量检查 Gate

### 新增文档发布前必须通过：

1. [ ] Frontmatter 完整且格式正确
2. [ ] 所有 wikilink 可解析
3. [ ] `routing_target` 在 JSON 路由文件中有对应条目
4. [ ] 内容无编造数据（实测数据必须标注来源）
5. [ ] 包含至少 1 条可操作 Checklist 或 SOP 步骤
6. [ ] `decision_value` 描述决策价值（不是数字）
7. [ ] JSON 路由文件校验通过
