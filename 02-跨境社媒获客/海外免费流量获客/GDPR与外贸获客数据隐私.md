# GDPR 与外贸获客数据隐私

> **适用范围**：所有通过社媒/邮件/WhatsApp 等渠道获取和处理欧盟居民个人数据的外贸企业
> **核心原则**：GDPR 不仅适用于欧盟境内的企业。只要你的业务涉及处理欧盟居民的个人数据，无论你的公司在哪里注册，都受 GDPR 约束。
> **免责声明**：本文档提供的是基于公开信息的一般性合规指南，NOT法律建议。建议在开展涉及欧盟客户数据的业务前，咨询数据隐私法律专业人士。

---

## 一、GDPR 适用范围

### 1.1 谁受 GDPR 约束

GDPR 适用于以下两种情况之一：

1. **机构设立标准**（Article 3(1)）：企业在欧盟境内有机构，且数据处理活动与该机构的活动相关
2. **目标指向标准**（Article 3(2)）：企业不在欧盟境内，但向欧盟居民提供商品/服务或监控其行为

**外贸企业何时受约束**：

| 场景 | 是否受 GDPR 约束 |
|------|-----------------|
| 通过 LinkedIn 联系欧盟采购商 | 是（向他们提供服务） |
| 在 Facebook 上向欧盟用户投放广告 | 是（监控行为/目标指向） |
| 收集欧盟客户的邮箱发送产品报价 | 是（处理个人数据） |
| 公司网站有 Cookie 追踪欧盟访客 | 是（监控行为） |
| CRM 中存储了欧盟客户的联系方式 | 是（存储个人数据） |
| 只与非欧盟客户做生意 | 否 |

### 1.2 关键概念

- **数据控制者（Controller）**：决定数据处理目的和方式的实体（通常是你的公司）
- **数据处理者（Processor）**：代表控制者处理数据的实体（如 CRM 提供商、邮件营销平台）
- **个人数据（Personal Data）**：与已识别或可识别的自然人相关的任何信息
- **处理（Processing）**：对个人数据进行的任何操作（收集、存储、使用、传输、删除等）

**来源**：
- GDPR Article 3（地域范围）：https://gdpr-info.eu/art-3-gdpr/
- GDPR 全文：https://gdpr-info.eu/

## 二、个人数据定义

### 2.1 什么构成个人数据

GDPR 下的个人数据包括**任何能直接或间接识别自然人的信息**。

**B2B 场景中的个人数据**：

| 数据类型 | 是否个人数据 | 说明 |
|---------|-------------|------|
| 个人姓名 | 是 | 直接可识别 |
| 工作邮箱（含姓名，如 john@company.com） | 是 | 可识别到个人 |
| 个人邮箱 | 是 | 明确是个人数据 |
| 手机号 | 是 | 直接可识别 |
| 公司名称 | 否 | 法人实体，NOT自然人 |
| 通用邮箱（如 info@company.com） | 通常否 | 不指向特定个人 |
| LinkedIn 个人主页 URL | 是 | 可识别到个人 |
| 职位/头衔 | 与姓名结合时是 | 单独不构成 |
| IP 地址 | 是 | 欧盟法院认定为个人数据 |
| Cookie ID | 是 | 欧盟法院认定为个人数据 |

### 2.2 B2B 联系人信息是否包含在内

**关键结论**：**是的，B2B 联系人信息（采购经理/老板/技术人员的姓名、邮箱、电话等）属于个人数据**，即使在 B2B 场景中也是如此。

GDPR 保护的是**自然人（individual）**的数据，NOT法人实体（company）。但 B2B 联系人是自然人，他们的职业联系信息同样受 GDPR 保护。

**欧盟各国差异**：
- 部分国家（如德国）对 B2B 营销有额外限制
- 部分国家允许基于"合法利益"向企业邮箱发送 B2B 营销邮件
- 但最佳实践是：统一按 GDPR 标准处理所有个人数据

**来源**：
- GDPR Article 4(1)（个人数据定义）：https://gdpr-info.eu/art-4-gdpr/
- EDPB（欧洲数据保护委员会）指南：https://edpb.europa.eu/

## 三、合法处理依据

### 3.1 六种合法依据（Article 6）

处理个人数据必须基于以下六种合法依据之一：

| 合法依据 | 说明 | B2B 获客适用性 |
|---------|------|--------------|
| **同意（Consent）** | 数据主体明确同意 | 适用，但需满足严格的同意标准 |
| **合同必要性** | 为履行合同所必需 | 不适用于获客阶段 |
| **法律义务** | 为履行法律义务所必需 | 不适用于获客 |
| **重大利益** | 为保护数据主体的重大利益 | 不适用于获客 |
| **公共利益** | 为执行公共利益任务 | 不适用于获客 |
| **合法利益（Legitimate Interest）** | 控制者的合法利益，不凌驾于数据主体的权利和自由 | **最常用的 B2B 获客依据** |

### 3.2 合法利益（Legitimate Interest）详解

**什么是合法利益**：

你的公司作为数据控制者，有合法的商业利益去处理个人数据（如 B2B 营销），但这种利益不能凌驾于数据主体的权利和自由之上。

**合法利益测试（三要素测试）**：

1. **目的测试（Purpose Test）**：你是否有合法的商业利益？
   - B2B 营销通常满足（推广产品/服务给潜在客户是正当商业行为）

2. **必要性测试（Necessity Test）**：处理数据对实现该目的是否必要？
   - 你是否可以不用个人数据就达到同样目的？如果可以，则不满足

3. **平衡测试（Balancing Test）**：你的利益是否凌驾于数据主体的权利和自由之上？
   - 需要考虑：数据类型、数据主体的合理期望、处理的影响、保护措施

**合法利益 vs 同意——B2B 获客场景对比**：

| 维度 | 合法利益 | 同意 |
|------|---------|------|
| 适用场景 | 现有/潜在客户 B2B 联系 | 网站注册/邮件订阅 |
| 实施难度 | 中（需做 LIA 评估） | 低（但需维护同意记录） |
| 灵活性 | 较高 | 较低（同意可随时撤回） |
| 风险 | 需能证明通过了三要素测试 | 同意标准严格，需证明 |
| 推荐用途 | 冷邮件/LinkedIn 联系 | 网站表单/订阅列表 |

### 3.3 合法利益评估（LIA）实操

**LIA 简版模板**：

```
1. 目的：我们处理 [数据类型] 是为了 [具体目的]
2. 必要性：处理这些数据是实现目的所必要的，因为 [原因]
3. 平衡：
   - 数据类型：[低敏感度——工作联系信息]
   - 数据主体期望：[B2B 场景中接收相关商业信息是可预期的]
   - 影响：[最小——提供退订选项]
   - 保护措施：[隐私政策/退订/数据安全]
4. 结论：合法利益成立
```

**注意**：
- 如果处理的是敏感数据（健康、种族、政治等），合法利益不适用
- 欧盟某些成员国（如德国）对 B2B 营销有额外的法律限制
- 如果数据主体反对（opt-out），必须立即停止处理

**来源**：
- GDPR Article 6（合法处理）：https://gdpr-info.eu/art-6-gdpr/
- ICO（英国信息专员办公室）合法利益指南：https://ico.org.uk/for-organisations/guide-to-data-protection/guide-to-the-general-data-protection-regulation-gdpr/lawful-basis-for-processing/legitimate-interests/

## 四、数据收集合规要求

### 4.1 透明度要求（Article 12-14）

当你从不同来源收集个人数据时，需要向数据主体提供以下信息：

**直接从数据主体收集时（如网站表单）**：
- 你的身份和联系方式
- 数据处理的目的和法律依据
- 数据的接收者（如有）
- 数据保留期限
- 数据主体的权利（访问、更正、删除等）
- 是否向第三国传输数据

**间接收集时（如从 LinkedIn/展会名单获取）**：
- 以上所有信息
- 此外，还需说明数据来源于何处
- 必须在"合理期限内"（通常是一个月内）提供这些信息

### 4.2 数据最小化（Article 5(1)(c)）

- 只收集实现目的所必需的最少数据
- 不要收集"可能以后有用"的数据
- B2B 获客：姓名、工作邮箱、公司、职位通常足够

### 4.3 目的限制（Article 5(1)(b)）

- 只能将数据用于收集时声明的目的
- 不能将获客收集的数据用于其他未声明的用途
- 如需改变用途，需重新获得法律依据

### 4.4 社媒获客中的合规要点

| 场景 | 合规要求 |
|------|---------|
| 从 LinkedIn 获取联系人信息 | 记录来源；基于合法利益或同意；提供隐私政策链接 |
| 通过 Facebook 表单收集询盘 | 表单中必须有隐私政策链接；明确说明数据用途 |
| WhatsApp 对话中获取信息 | 对话开始时说明数据处理方式；提供退订选项 |
| 展会名片收集 | 展会后尽快发送隐私政策；说明数据用途 |
| 网站访客追踪（Cookie） | Cookie 同意弹窗；隐私政策说明 |

## 五、WhatsApp / 邮箱营销的 GDPR 合规

### 5.1 WhatsApp 营销

**核心原则**：WhatsApp 消息包含个人数据（手机号、消息内容），受 GDPR 约束。

**合规要求**：
1. **法律依据**：需要数据主体的同意或合法利益
   - 对于**首次冷联系**（对方从未与你沟通过），建议使用**同意**作为法律依据
   - 对于**现有客户/已有业务关系**，可考虑**合法利益**

2. **隐私通知**：在首次联系时提供隐私政策链接，说明：
   - 你是谁
   - 为什么联系他们
   - 如何处理他们的数据
   - 如何退订

3. **退选权**：必须提供简单的退选方式
   - 回复 "STOP" 或类似关键词即可退订
   - 退选后必须立即停止联系并删除数据

4. **数据最小化**：只保存必要的联系人信息

5. **ePrivacy 指令补充**：欧盟的 ePrivacy 指令（各国实施不同）对电子通信营销有额外要求
   - 多数欧盟国家要求对个人的电子营销必须有**事先同意**（opt-in）
   - B2B 场景下部分国家允许基于 opt-out

**WhatsApp 冷联系建议话术**：
```
Hello [Name], this is [Your Name] from [Company]. We are a manufacturer
of [product category]. I found your contact through [source, e.g.,
LinkedIn / your company website]. I'm reaching out because [specific
reason related to their business]. If you'd prefer not to receive
messages from us, please reply "STOP" and I will remove your contact
immediately. Our privacy policy: [link]
```

### 5.2 邮箱营销

**核心要求**：

1. **合法依据**：
   - B2B 邮箱：多数欧盟国家允许基于合法利益（但各国实施不同）
   - B2C 邮箱：必须基于明确的事先同意（opt-in）

2. **CAN-SPAM（美国客户）**：
   - 真实发件人信息
   - 不误导的主题行
   - 包含退订链接
   - 包含公司物理地址
   - 10 个工作日内处理退订

3. **邮件内容合规**：
   - 清晰的发件人身份
   - 明确的退订方式
   - 隐私政策链接
   - 真实准确的产品信息

4. **退选管理**：
   - 每封邮件必须包含退订链接
   - 退选请求必须在合理时间内处理
   - 退选后的联系人不得再次加入营销列表

### 5.3 各国对 B2B 邮件营销的额外要求

| 国家 | B2B 邮件要求 |
|------|-------------|
| 德国 | 严格——原则上需要事先同意，除非有"现有客户关系" |
| 法国 | 允许 B2B 冷邮件，但需提供退订选项 |
| 英国 | 允许 B2B 冷邮件给公司邮箱（非个人邮箱），需退订选项 |
| 荷兰 | 允许 B2B 冷邮件，需退订选项 |
| 西班牙 | 需要事先同意（包括 B2B） |
| 意大利 | 需要事先同意（包括 B2B） |

**建议**：以最严格的国家（德国、西班牙、意大利）为标准，统一要求事先同意，可以最大程度降低风险。

## 六、CRM 数据存储要求

### 6.1 安全要求（Article 32）

GDPR 要求数据控制者采取适当的技术和组织措施保护个人数据安全：

**技术要求**：
- 数据加密（存储和传输）
- 访问控制（只有授权人员可访问）
- 定期安全测试
- 备份和恢复能力

**组织要求**：
- 员工数据保护培训
- 访问权限管理（最小权限原则）
- 数据处理记录
- 安全事件应急流程

### 6.2 CRM 选择建议

选择符合 GDPR 的 CRM 系统时，检查以下项目：

| 检查项 | 说明 |
|--------|------|
| 数据处理协议（DPA） | CRM 提供商是否提供 DPA |
| 数据存储位置 | 数据存储在哪些国家/地区 |
| 标准合同条款（SCC） | 如果数据传出欧盟/EEA，是否有 SCC |
| 数据删除能力 | 是否可以完全删除数据（满足被遗忘权） |
| 数据导出能力 | 是否可以导出数据（满足数据可携权） |
| 访问日志 | 是否记录谁在什么时候访问了什么数据 |
| 加密 | 数据是否加密存储和传输 |

**常见 GDPR 合规 CRM**：
- HubSpot（提供 GDPR 合规工具）
- Salesforce（提供 GDPR 合规功能）
- Pipedrive（欧盟公司，GDPR 合规）
- Zoho CRM（提供 GDPR 合规功能）

### 6.3 数据保留期限

- GDPR 没有规定具体的数据保留期限
- 原则：数据保留不应超过实现目的所需的时间
- 建议：
  - 活跃客户数据：业务关系存续期间 + 合理期限（如 3 年）
  - 未成交线索：12-24 个月后评估是否删除
  - 已退订联系人：保留"已退订"状态（防止误发），但删除其他个人数据
  - 建立定期数据清理流程（如每年审查一次）

## 七、跨境数据传输规则

### 7.1 基本原则

- 个人数据可以在欧盟/EEA 内自由流动
- 向欧盟/EEA 以外传输数据需要满足特定条件

### 7.2 合法传输机制

| 机制 | 说明 | 适用场景 |
|------|------|---------|
| **充分性认定（Adequacy Decision）** | 欧盟认定该国家/地区提供充分的数据保护 | 传输到日本、韩国、英国、阿根廷等 |
| **标准合同条款（SCCs）** | 使用欧盟委员会批准的标准合同条款 | 最常用的传输机制 |
| **约束性企业规则（BCRs）** | 跨国集团内部的数据传输规则 | 大型跨国企业 |
| **明确同意** | 数据主体在知情后明确同意 | 一次性/偶尔传输 |

### 7.3 中国企业的实际情况

- 中国目前**未被**欧盟认定为具有"充分保护水平"
- 中国企业处理欧盟客户数据时，需要使用 SCCs 或其他合法传输机制
- 如果使用云服务（AWS、阿里云等），需确认数据存储位置和传输机制

### 7.4 实操建议

1. **选择数据存储位置**：优先选择欧盟境内的服务器存储欧盟客户数据
2. **签署 SCCs**：与数据接收方（如 CRM 提供商、云服务商）签署标准合同条款
3. **传输影响评估（TIA）**：评估数据接收国的法律环境是否影响数据保护
4. **隐私政策披露**：在隐私政策中说明数据的跨境传输情况
5. **监控法规变化**：充分性认定和 SCCs 可能随法规变化而更新

**来源**：
- GDPR 第五章（跨境传输）：https://gdpr-info.eu/chapter-5/
- 欧盟委员会 SCCs：https://commission.europa.eu/law/law-topic/data-protection/international-dimension-data-protection-eu/standard-contractual-clauses-scc_en
- 充分性认定列表：https://commission.europa.eu/law/law-topic/data-protection/international-dimension-data-protection-eu/adequacy-decisions_en

## 八、违规处罚

### 8.1 罚款等级

| 等级 | 上限 | 适用违规类型 |
|------|------|-------------|
| 一级 | 1000 万欧元或全球年营业额 2%（取高者） | 技术/组织措施不足；DPO 相关；认证相关 |
| 二级 | 2000 万欧元或全球年营业额 4%（取高者） | 违反处理原则（合法依据、同意、数据主体权利等）；跨境传输违规 |

### 8.2 实际处罚案例

- Google：5000 万欧元（法国 CNIL，透明度与同意问题）
- Amazon：7.46 亿欧元（卢森堡，定向广告合法性）
- Meta（Facebook）：多次被罚，累计数亿欧元
- British Airways：2200 万英镑（数据泄露）
- WhatsApp：2.25 亿欧元（爱尔兰，透明度问题）

**注意**：对于小型外贸企业，监管机构更倾向于先发出警告和整改通知，而非直接罚款。但如果收到整改通知后不配合，将面临罚款。

### 8.3 其他后果

- 声誉损失
- 客户要求删除数据
- 业务合作终止
- 民事诉讼（数据主体可索赔）

## 九、实操检查清单

### 9.1 基础合规（必须完成）

- [ ] **隐私政策**：官网/着陆页有 GDPR 合规的隐私政策
- [ ] **合法依据**：确定每个数据处理活动的合法依据并记录
- [ ] **同意管理**：网站表单有明确的 opt-in（非预选勾选项）
- [ ] **Cookie 同意**：有 GDPR 合规的 Cookie 同意弹窗
- [ ] **退选机制**：所有营销邮件/消息包含退订方式
- [ ] **数据记录**：记录你收集了哪些人的哪些数据、来源、用途
- [ ] **CRM 合规**：确认 CRM 系统符合 GDPR 要求

### 9.2 进阶合规（强烈建议）

- [ ] **合法利益评估（LIA）**：如果基于合法利益处理数据，完成并保存 LIA 文档
- [ ] **数据处理协议（DPA）**：与所有处理你客户数据的第三方签署 DPA
- [ ] **数据跨境传输机制**：确认数据传输到中国的合法机制（如 SCCs）
- [ ] **数据主体请求流程**：建立处理访问/删除/更正请求的流程
- [ ] **数据保留政策**：制定并执行数据清理计划
- [ ] **员工培训**：确保处理客户数据的员工了解 GDPR 基本要求
- [ ] **安全事件应急**：制定数据泄露应急响应计划

## 十、常见问题

### Q1：我只联系欧盟企业的采购部门，不涉及个人，是否不需要遵守 GDPR？

**答**：否。采购经理/联系人是自然人，他们的姓名、邮箱、电话属于个人数据，受 GDPR 保护。即使是在 B2B 场景中也是如此。

### Q2：我从 LinkedIn 上公开可见的资料获取联系方式，是否还需要法律依据？

**答**：是。数据是否公开不影响 GDPR 的适用。即使数据是公开的，处理（收集、存储、使用）这些数据仍需要合法依据。

### Q3：合法利益和同意，B2B 获客应该选哪个？

**答**：两者都可以，但适用场景不同：
- **同意**更适合：网站注册、邮件订阅、展会主动交换名片等
- **合法利益**可考虑：冷邮件 B2B 联系（但需完成 LIA 评估并提供 opt-out）
- **最安全做法**：尽可能获取明确同意

### Q4：客户退订后，我需要删除所有数据吗？

**答**：退订意味着停止营销用途的处理。你可以保留"此人已退订"的记录（防止误发），但应删除其他个人数据。如果数据还有其他合法用途（如合同关系），可以继续保留。

### Q5：GDPR 违规一定会被罚吗？

**答**：不一定。监管机构通常先发出警告和整改通知。但收到通知后必须认真整改，否则将面临罚款。罚款金额取决于违规的严重程度、是否配合、是否有先例等。
