# AI 用量审计与成本优化报告

## 结论

本机 AI 成本的主要问题不是“平台太多”，而是：

1. Claude / Codex / OpenClaw / cc-connect / 飞书桥接并行，模型入口分散。
2. Claude 历史用量里 cache read 极大，说明长会话和大上下文反复复用，是主要成本与延迟来源。
3. 搜索类 MCP 已被分层，但 cc-connect 旧长会话仍带 Tavily / Serper / Lark 子进程。
4. 用量统计系统已存在，但当前统计文件为空或未持续采集，无法直接对账到真实账单。

## 当前价格基准

- OpenAI GPT-5.5：input $5/M、cached input $0.5/M、output $30/M；Web search $10/1k calls。
- Claude Sonnet 4.6：input $3/M、cache hit $0.30/M、output $15/M。
- Qwen qwen3.6-plus：≤256K 请求 input $0.50/M、output $3/M；256K-1M input $2/M、output $6/M。
- MiniMax-M2.7：input $0.3/M、output $1.2/M、cache read $0.06/M、cache write $0.375/M。
- MiniMax-M2.7-highspeed：input $0.6/M、output $2.4/M、cache read $0.06/M、cache write $0.375/M。
- Tavily：每月 1000 free credits；PAYG $0.008/credit；basic search 1 credit，advanced search 2 credits。
- Firecrawl：每月 1000 free credits；search 1 credit/result，scrape 1 credit/page，interact 5 credits/action。
- OpenRouter：Pay-as-you-go 平台费 5.5%，BYOK 超过 1M requests/月后 5% fee。

## 立刻执行的优化建议

### P0：不影响能力，先止血

1. 保持 Claude MCP 默认 `core`，只启用 `claude-mem`。
2. cc-connect 安排一次温和重启，让旧会话不再常驻 Tavily / Serper / Lark。
3. 飞书桥接默认模型不要用 GPT-5.5；普通消息用 MiniMax-M2.7 或 Qwen，只有高价值任务手动升级。
4. 搜索统一优先 Tavily 或 Serper 之一，避免同一问题双搜。

### P1：路由策略

| 任务 | 推荐默认 | 升级条件 |
|---|---|---|
| 日常编码/配置 | qwen3.6-plus | 复杂架构或疑难 bug 再升级 |
| 飞书/Telegram 普通答复 | MiniMax-M2.7 | 用户明确要求高质量长文 |
| 多角色圆桌 | MiniMax-M2.7 | 需要快响应再 highspeed |
| 搜索研究 | Tavily basic | 竞品/深度报告再 advanced/research |
| 网页抓取 | Firecrawl scrape | 大规模 crawl 先列 URL 白名单 |
| 高价值代码审查 | Claude Sonnet / GPT-5.5 | 仅在关键项目、最终验收使用 |

### P2：建立真实账单闭环

1. 修复现有 `data/usage_stats` 采集，让 cc-connect、feishu-codex-bridge、Claude Code Router 都写入同一个 usage ledger。
2. 给每个 API Key 建预算标签：`personal-coding`、`bridge`、`research`、`marketing`。
3. 每周生成一页成本报告：模型、项目、token、搜索 credits、估算美元、异常请求。

## 最可能省钱的三个动作

1. 重启 cc-connect 并继承 MCP core profile，减少搜索/Lark MCP 常驻。
2. 把飞书桥接的 `CODEX_DEFAULT_MODEL=gpt-5.5` 改成低成本默认，高价值请求再升级。
3. 对 Obsidian 大规模治理任务采用“索引摘要 -> 小批执行 -> 最终验收”三段式，避免反复把超大上下文带进每轮对话。
