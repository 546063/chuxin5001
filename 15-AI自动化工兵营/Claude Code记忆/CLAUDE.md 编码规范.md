# CLAUDE.md 编码规范

> 减少 LLM 编码常见错误的行为准则。可与项目具体需求合并使用。

---

## 1. 编码前先思考

**不假设、不藏困惑、主动暴露权衡。**

- 明确说出假设。如果不确定，就问。
- 多种解读时全部列出——不默默选一个。
- 有更简单的方案直接说出来，该反驳就反驳。
- 不清楚的地方停下来，说出哪里困惑，然后提问。

## 2. 简单优先

**只写解决问题所需的最少代码。**

- 不添加需求之外的功能。
- 不为一次性代码创建抽象。
- 不添加"灵活性"或"可配置性"。
- 不处理不可能发生的错误场景。
- 能写 50 行就别写 200 行，重写。

检验："高级工程师会觉得这太复杂吗？"如果是，简化。

## 3. 精准改动

**只改必须改的，不顺手优化无关代码。**

- 不"顺便优化"相邻代码、注释或格式。
- 不重构没坏的东西。
- 匹配已有风格。
- 无关的死代码提一下，别删。
- 自己改动产生的孤立代码要清理。

检验：每一行变更都能追溯到用户需求。

## 4. 目标驱动执行

**成功标准要可验证，循环验证直到确认。**

| 任务类型 | 成功标准 |
|---------|---------|
| "加个校验" | 先写无效输入的测试，然后让它们通过 |
| "修掉这个 bug" | 先写能复现的测试，然后让测试通过 |
| "重构 X" | 确保重构前后测试都通过 |

多步骤任务列出简要计划：
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
